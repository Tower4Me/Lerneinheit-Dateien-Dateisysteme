Modul 01 · Bits & Bytes

Bits & Bytes

Alles was ein Computer speichert, verarbeitet oder überträgt, besteht auf dem untersten Level aus zwei Zuständen: Spannung an oder aus. Das sind 1 und 0. Darauf baut alles auf.

Das Bit

Ein Bit (Binary Digit = binäre Stelle) ist die kleinste mögliche Informationseinheit. Es kann genau zwei Zustände annehmen: 1 oder 0. An oder aus. Wahr oder falsch.

Ein einzelnes Bit ist für sich wenig nützlich. Erst durch Kombination mehrerer Bits entstehen aussagekräftige Werte.

Bit

1 oder 0

Kleinste Einheit. Kommt von "binary digit". Ein Bit kann nur zwei verschiedene Werte haben.

Nibble

4 Bits

Halb so groß wie ein Byte. Kann Werte von 0 bis 15 darstellen. Entspricht genau einer Hex-Stelle.

Byte

8 Bits

Die grundlegende Speichereinheit. Kann 28 = 256 verschiedene Werte annehmen (0 bis 255).

Kilobyte (KB)

1.024 Byte

Im binären System: 210 = 1.024 Byte. Im Dezimalsystem (Festplattenhersteller): 1.000 Byte.

Warum 8 Bits pro Byte?

Das ist historisch gewachsen. Frühe Computer nutzten 6 oder 7 Bits. IBM setzte 1964 mit dem System/360 den Standard auf 8 Bits, weil das reicht um alle druckbaren Zeichen plus Steuerzeichen darzustellen und gut durch 2 teilbar ist.

Was kann ein Byte darstellen?

Mit 8 Bits sind genau 256 Kombinationen möglich (2 hoch 8). Das reicht für:

Beispiel01000101 = 0x65 = 'e' = Dezimal 101 0x bedeutet: diese Zahl ist hexadezimal (Basis 16) angegeben. 0x65 ist nicht "null x 65", sondern das Präfix für eine Hex-Zahl. 6 * 16 + 5 = 101 -- das ist derselbe Wert wie Dezimal 101.
0x erklärt

0x ist ein Präfix das anzeigt: "die folgende Zahl ist hexadezimal". 0x65 bedeutet also: Hex-Zahl 65. Das entspricht Dezimal 101. Genauso zeigt 0b eine Binärzahl an: 0b01000101 = dieselbe 101.

Zahlensysteme

Computer rechnen binär, Menschen rechnen dezimal, und Entwickler lesen Hex. Alle drei Systeme beschreiben dieselben Werte – nur mit unterschiedlichen Symbolen.

Drei Systeme, ein Wert

Dezimal (Basis 10)

0–9

Das System das Menschen seit Jahrtausenden nutzen. 10 Symbole, weil wir 10 Finger haben. Stellenwert: 1, 10, 100, 1000...

Binär (Basis 2)

0 und 1

Das System der Computer. Nur 2 Symbole, weil Hardware nur zwei Zustände kennt. Stellenwert: 1, 2, 4, 8, 16...

Hexadezimal (Basis 16)

0–9 und A–F

16 Symbole. A=10, B=11, ... F=15. Sehr kompakt: 2 Hex-Stellen = 1 Byte. Deshalb überall in der IT.

Beispiel: Wert 255Dezimal: 255 Binär: 11111111 -- 8 Bits, alle auf 1 Hexadezimal: 0xFF -- F=15, also 15*16 + 15 = 255 Dezimal: 65 Binär: 01000001 -- 64+1 = 65 Hexadezimal: 0x41 -- 4*16 + 1 = 65 = 'A' in ASCII

Wie liest man Binär?

Jede Stelle hat einen Stellenwert der sich verdoppelt: von rechts nach links 1, 2, 4, 8, 16, 32, 64, 128. Du addierst die Werte der Stellen die eine 1 haben.

Beispiel: 0b10110010Stelle: 128 64 32 16 8 4 2 1 Bit: 1 0 1 1 0 0 1 0 Wert: 128 + 0 + 32 + 16 + 0 + 0 + 2 + 0 = 178

Wie liest man Hexadezimal?

Jede Hex-Stelle entspricht genau 4 Bits (einem Nibble). Das macht Hex so praktisch: ein Byte = 2 Hex-Stellen, immer.

Hex lesen0xB2 = B * 16 + 2 = 11 * 16 + 2 = 176 + 2 = 178 -- Schnelle Umrechnung: jede Hex-Stelle = 4 Bits B = 11 = 1011 2 = 2 = 0010 B2 = 10110010 = 0b10110010 = 178

Modul 02 · Was ist eine Datei?

Dateien & Metadaten

Eine Datei ist eine benannte, geordnete Folge von Bytes, dauerhaft auf einem Speichermedium abgelegt. Das klingt simpel – aber dahinter steckt ein ganzes System.

Was ist eine Datei?

Auf dem untersten Level ist eine Datei nichts anderes als eine Sequenz von Bytes auf einem Speichermedium. Das Betriebssystem (Operating System = OS) weiß nicht was in der Datei steckt. Es verwaltet nur wo sie liegt und wie sie heißt. Den Inhalt interpretiert immer das öffnende Programm.

Wichtig

Die Dateiendung (z.B. .jpg) ist nur ein Hinweis für das Betriebssystem welches Programm die Datei öffnen soll. Sie kann umbenannt werden ohne den Inhalt zu verändern. Der echte Dateityp steckt in den ersten Bytes der Datei selbst (dazu mehr in Modul 4).

Metadaten

Metadaten (Meta = über, Daten = Informationen) sind Daten über eine Datei, nicht der Inhalt selbst. Das Dateisystem speichert sie getrennt vom eigentlichen Dateiinhalt.

Name

Dateiname + Endung

Der menschenlesbare Name. Wird vom Dateisystem gespeichert, nicht in der Datei selbst.

Größe

Dateigröße in Bytes

Wie viele Bytes der Inhalt hat. Unterschied zur "Größe auf Datenträger" (immer ein Vielfaches der Clustergröße).

Zeitstempel

Erstellt / Geändert / Geöffnet

Drei verschiedene Zeitstempel: Erstellung, letzte Änderung, letzter Zugriff. Forensisch relevant.

Berechtigungen

Wer darf was

Lesen, Schreiben, Ausführen. Unter Linux rwx, unter Windows ACL (Access Control List). Mehr dazu in Modul 6.

Speicherort

Inode / MFT-Eintrag

Zeiger auf die tatsächlichen Datenblöcke auf dem Datenträger. Der "Inhaltsverzeichnis-Eintrag" der Datei.

Attribute

Versteckt, System, Archiv

Zusätzliche Flags. Z.B. "versteckt" unter Windows, "system" für Betriebssystemdateien.

Inode – der interne Datei-Eintrag (Linux)

Unter Linux verwaltet das Dateisystem jede Datei über einen Inode (Index Node = Index-Knoten). Der Inode enthält alle Metadaten außer dem Dateinamen. Der Dateiname ist nur ein Verweis auf den Inode.

Linux Terminal# Inode-Nummer einer Datei anzeigen ls -i datei.txt # Ausgabe: 1234567 datei.txt # Alle Metadaten einer Datei anzeigen stat datei.txt # Zeigt: Größe, Blöcke, Zugriffsrechte, Inode, Zeitstempel
Warum ist das relevant?

Wenn du eine Datei "löschst", wird meist nur der Inode-Eintrag als frei markiert. Die eigentlichen Daten bleiben auf dem Datenträger bis sie überschrieben werden. Das ist der Grund warum gelöschte Dateien mit Forensik-Tools wiederhergestellt werden können.

MFT – Master File Table (Windows/NTFS)

Windows NTFS (New Technology File System) nutzt statt Inodes die MFT (Master File Table = Haupt-Dateitabelle). Jede Datei hat einen MFT-Eintrag mit allen Metadaten. Bei kleinen Dateien (< ~700 Byte) wird der Inhalt direkt im MFT-Eintrag gespeichert.

Größe vs. Größe auf Datenträger

Windows zeigt zwei Größen: "Größe" ist die tatsächliche Dateigröße in Bytes. "Größe auf Datenträger" ist immer ein Vielfaches der Clustergröße (z.B. 4 KB). Eine 1-Byte-Datei belegt trotzdem 4 KB auf der Festplatte. Der Rest ist verschwendeter Platz (Slack Space = Leerraum).

Modul 02 · Was ist eine Datei?

Dateisysteme

Ein Dateisystem ist die Struktur die ein Betriebssystem nutzt um Daten auf einem Speichermedium zu organisieren. Ohne Dateisystem wäre eine Festplatte nur ein riesiger Haufen Bytes ohne Anfang und Ende.

Was macht ein Dateisystem?

Ein Dateisystem löst vier Grundprobleme:

Organisation

Ordner & Dateien

Hierarchische Struktur aus Verzeichnissen und Dateien. Wo fängt was an, wo hört es auf.

Adressierung

Wo liegt was

Welche Blöcke auf dem Datenträger gehören zu welcher Datei. Zuordnung von Namen zu Speicherorten.

Freispeicher

Was ist noch frei

Verwaltung welche Blöcke belegt und welche frei sind. Ohne das könnte man nichts schreiben.

Metadaten

Infos über Dateien

Name, Größe, Zeitstempel, Berechtigungen – alles was nicht der Dateiinhalt selbst ist.

Die wichtigsten Dateisysteme

NTFS – New Technology File System

Das Standard-Dateisystem von Windows seit Windows NT (1993). Heute auf praktisch allen Windows-Systemen.

Stärken
  • Dateigrößen bis 16 EB (Exabyte) theoretisch
  • Journaling (Fehlertoleranz bei Stromausfall)
  • Zugriffsrechte (ACL = Access Control List)
  • Verschlüsselung (EFS = Encrypting File System)
  • Komprimierung einzelner Dateien
Schwächen
  • Nur von Linux mit zusätzlichem Treiber beschreibbar
  • macOS kann nur lesen, nicht schreiben
  • Für USB-Sticks überdimensioniert

FAT32 – File Allocation Table 32

Ein altes aber universelles Dateisystem (1996). Wird heute vor allem für USB-Sticks und SD-Karten genutzt wegen maximaler Kompatibilität.

Kritische Einschränkung

FAT32 unterstützt keine einzelne Datei größer als 4 GB (genau: 4 GB minus 1 Byte). Das ist ein hartes Limit im Dateisystemdesign. Ein 8 GB großes Video passt deshalb nicht auf einen FAT32-formatierten USB-Stick.

Stärken
  • Von Windows, Linux, macOS, Kameras, Radios lesbar
  • Einfach und robust
  • Geringe Overhead-Größe
Schwächen
  • Max. 4 GB pro Datei
  • Max. 32 GB Partitionsgröße (Windows-Formatierung)
  • Kein Journaling, keine Berechtigungen

exFAT – Extended File Allocation Table

Microsofts Nachfolger für FAT32 (2006). Keine 4-GB-Grenze mehr, trotzdem breit kompatibel. Heute Standard für große SD-Karten (SDXC).

ext4 – Fourth Extended Filesystem

Das Standard-Dateisystem von Linux seit 2008. Robust, schnell, mit Journaling.

Stärken
  • Sehr zuverlässig und erprobt
  • Journaling, große Dateien, große Partitionen
  • Standardmäßig auf fast allen Linux-Systemen
Schwächen
  • Windows kann ext4 nicht lesen (ohne Drittanbieter-Software)
  • macOS auch nicht nativ

Was ist Journaling?

Journaling (Journal = Tagebuch) bedeutet: bevor das Dateisystem eine Änderung vornimmt, schreibt es sie zuerst in ein Journal. Wenn der Strom während einer Schreiboperation ausfällt, kann das Dateisystem beim nächsten Start das Journal durchgehen und unvollständige Operationen rückgängig machen. Ohne Journaling können Dateien nach einem Absturz korrumpiert sein.

Modul 02 · Was ist eine Datei?

Journaling & Fragmentierung

Zwei Konzepte die direkt die Zuverlässigkeit und Geschwindigkeit eines Dateisystems bestimmen. Beide sind IHK-relevant.

Journaling

Stell dir vor du schreibst eine Datei und mitten im Schreibvorgang fällt der Strom aus. Ohne Sicherheitsmechanismus liegt die Datei jetzt halb geschrieben auf der Platte: der Verzeichniseintrag sagt die Datei existiert, aber die Daten sind unvollständig. Das Dateisystem ist inkonsistent.

Journaling erklärt

Journaling (Journal = Tagebuch/Protokoll) löst dieses Problem. Bevor das Dateisystem eine Änderung vornimmt, schreibt es die geplante Aktion zuerst in ein separates Journal (einen reservierten Bereich auf dem Datenträger). Erst danach wird die eigentliche Änderung ausgeführt. Nach einem Absturz liest das System beim nächsten Start das Journal und weiß genau: welche Operationen waren abgeschlossen, welche nicht. Unvollständige werden rückgängig gemacht oder wiederholt.

Mit Journaling

NTFS, ext4, APFS

Nach einem Absturz: System prüft das Journal beim Booten, stellt konsistenten Zustand her. Geht schnell, meist sekunden. Keine korrumpierten Dateien.

Ohne Journaling

FAT32, exFAT

Nach einem Absturz: System muss den gesamten Datenträger prüfen (chkdsk / fsck). Dauert bei großen Laufwerken lange. Dateien können korrumpiert sein.

Alltagsvergleich

Journaling ist wie eine Einkaufsliste die du vor dem Einkaufen schreibst. Wenn du mitten im Laden ohnmächtig wirst, kann jemand anderes die Liste lesen und wissen: was ist schon im Korb, was fehlt noch. Ohne Liste weiß keiner wo du aufgehört hast.

Fragmentierung

Ein Dateisystem speichert Dateien in Blöcken (auch Cluster genannt). Wenn eine Datei zu groß für einen zusammenhängenden Bereich ist, werden ihre Blöcke verteilt gespeichert. Das nennt sich Fragmentierung.

Nicht fragmentiert

Blöcke liegen nebeneinander

Der Lesekopf einer Festplatte fährt einmal über den Bereich. Schnell. Bei SSDs (Solid State Drive = Flashspeicher) spielt die Position physikalisch keine Rolle.

Fragmentiert

Blöcke über die Platte verteilt

Der Lesekopf einer HDD (Hard Disk Drive = Festplatte mit Magnetscheibe) muss springen. Jeder Sprung kostet Zeit. Bei vielen kleinen Fragmenten wird die Platte messbar langsamer.

Warum fragmentiert ein Dateisystem?

Beim Schreiben sucht das Dateisystem freie Blöcke. Am Anfang liegen alles zusammenhängend. Mit der Zeit werden Dateien gelöscht, neue geschrieben, und die Freiflächen werden lückenhaft. Neue große Dateien passen nicht mehr in eine Lücke und werden aufgeteilt.

Defragmentierung

Defragmentierung (kurz: Defrag) ordnet die Blöcke neu an damit Dateien wieder zusammenhängend liegen. Windows hat seit XP einen integrierten Defragmentierer.

SSD niemals defragmentieren

SSDs haben keine beweglichen Teile. Fragmentierung verlangsamt SSDs kaum. Defragmentierung schreibt aber Daten um, was die begrenzte Schreiblebensdauer einer SSD unnötig verbraucht. Windows erkennt SSDs automatisch und deaktiviert die Defragmentierung für sie. Manuell defragmentieren ist bei SSDs kontraproduktiv.

Slack Space

Dateisysteme arbeiten in festen Blockgrößen (typisch 4 KB bei NTFS). Eine Datei die nur 1 KB groß ist, belegt trotzdem einen ganzen 4-KB-Block. Die restlichen 3 KB bleiben leer. Dieser verschwendete Platz heißt Slack Space.

Modul 02 · Was ist eine Datei?

Mounting & fstab

Unter Linux existiert keine Laufwerksbuchstaben wie C: oder D:. Stattdessen gibt es einen einzigen Verzeichnisbaum. Jedes Laufwerk wird in diesen Baum eingehängt.

Was ist Mounting?

Mounting (einhängen) bedeutet: ein Dateisystem wird einem Verzeichnis im bestehenden Verzeichnisbaum zugewiesen. Dieses Verzeichnis heißt Mountpoint (Einhängepunkt). Ab dem Moment des Mountens ist der Inhalt des Datenträgers über diesen Pfad erreichbar.

Alltagsvergleich

Stell dir vor du hast einen Aktenschrank (dein System) mit Schubladen (Verzeichnisse). Mounting ist wie das Einschieben einer neuen Schublade an einer bestimmten Position. Die Schublade existiert vorher, aber ihr Inhalt ist erst nach dem Einschieben erreichbar. Ziehst du sie heraus (unmount), ist der Inhalt wieder weg, aber die Schublade selbst ist unberührt.

Unter Windows vs. Linux

Linux
  • Ein einziger Baum, Wurzel ist /
  • Laufwerke werden in Verzeichnisse gemountet
  • Typisch: /mnt/usb oder /media/ben/stick
  • Manuell oder automatisch via /etc/fstab
Windows
  • Jedes Laufwerk hat einen Buchstaben: C:, D:, E:
  • Laufwerksbuchstaben werden automatisch vergeben
  • Auch Windows unterstützt Mountpoints (NTFS-Funktion)
  • Seltener manuell konfiguriert

Mount-Befehle unter Linux

Linux Terminal# Laufwerk manuell mounten mount /dev/sdb1 /mnt/usb # /dev/sdb1 = erste Partition des zweiten Datenträgers # /mnt/usb = Mountpoint (muss als Verzeichnis existieren) # Laufwerk wieder aushängen umount /mnt/usb # Achtung: kein "n" - es heisst umount, nicht unmount # Alle aktuell gemounteten Dateisysteme anzeigen mount # oder moderner: findmnt # Alle Laufwerke und Partitionen anzeigen lsblk

Die /etc/fstab

Die Datei /etc/fstab (File System Table = Dateisystemtabelle) steuert was beim Systemstart automatisch gemountet wird. Jede Zeile ist ein Eintrag für ein Dateisystem.

/etc/fstab – Aufbau# Gerät Mountpoint Typ Optionen dump pass /dev/sda1 / ext4 defaults 0 1 /dev/sda2 /home ext4 defaults 0 2 /dev/sdb1 /mnt/daten ntfs defaults 0 0 UUID=1A2B-3C4D /mnt/usb vfat auto,user 0 0 # UUID (Universally Unique Identifier) ist sicherer als /dev/sdX, # weil Gerätenamen sich ändern können.
Spalte 1

Gerät

Pfad wie /dev/sda1 oder besser UUID. Identifiziert den Datenträger.

Spalte 2

Mountpoint

Wohin wird gemountet. Muss ein existierendes Verzeichnis sein.

Spalte 3

Dateisystemtyp

ext4, ntfs, vfat (FAT32), exfat, nfs, cifs (SMB)...

Spalte 4

Optionen

defaults, ro (read-only), user (normale Nutzer dürfen mounten), noauto...

Fehler in fstab = System startet nicht

Eine fehlerhafte /etc/fstab kann dazu führen dass Linux beim Booten hängt. Deshalb immer vor der Änderung ein Backup machen und mit mount -a testen (versucht alle fstab-Einträge zu mounten ohne Neustart).

Modul 03 · Textdateien

ASCII & Zeichenkodierung

Wie weiß ein Computer welches Zeichen ein Byte darstellt? Durch Kodierungsstandards. Ohne gemeinsamen Standard würde jedes Programm Bytes anders interpretieren.

Das Problem

Ein Byte kann 256 verschiedene Werte haben (0–255). Aber welcher Wert entspricht welchem Buchstaben? Das ist nicht physikalisch vorgegeben, sondern eine Vereinbarung. Verschiedene Länder und Hersteller haben jahrzehntelang verschiedene Vereinbarungen getroffen, was zu Chaos geführt hat.

ASCII – American Standard Code for Information Interchange

ASCII (ausgesprochen "Ask-ee") ist der älteste weit verbreitete Standard (1963). Er definiert 128 Zeichen mit den Werten 0 bis 127 – also 7 Bit.

0 – 31

Steuerzeichen

Nicht druckbar. Z.B. 0x09 = Tab, 0x0A = Zeilenumbruch (LF), 0x0D = Wagenrücklauf (CR).

32 – 64

Sonderzeichen & Zahlen

Leerzeichen (32), !, ", #, $, %, &, Zahlen 0–9 (48–57), :, ;, <, =, >, ?, @

65 – 90

Großbuchstaben A–Z

A = 65, B = 66, ... Z = 90. Merkhilfe: 0x41 = 65 = 'A'. Hex und Dezimal.

97 – 122

Kleinbuchstaben a–z

a = 97, b = 98, ... z = 122. Genau 32 höher als die Großbuchstaben. Bit 5 macht den Unterschied.

ASCII BeispieleA = 65 = 0x41 = 01000001 a = 97 = 0x61 = 01100001 -- Bit 5 (32) ist gesetzt: 65 + 32 = 97 0 = 48 = 0x30 -- Achtung: Zeichen '0' != Zahl 0 = 32 = 0x20 -- Leerzeichen
Achtung: Zeichen vs. Zahl

Das Zeichen '0' hat den ASCII-Wert 48 (0x30). Die Zahl 0 hat den Wert 0. Das sind zwei völlig verschiedene Dinge. Ein häufiger Fehler beim Programmieren.

Das Problem mit ASCII: nur 128 Zeichen

ASCII deckt nur englische Zeichen ab. Kein Ä, kein Ö, kein Ü, kein ß, keine chinesischen Schriftzeichen, kein Arabisch. Jahrzehntelang hat jedes Land eigene Erweiterungen auf 256 Zeichen entwickelt (ISO-8859-1 für Westeuropa, Windows-1252 für Windows, etc.). Das führte zu Kompatibilitätsproblemen.

Unicode – ein Standard für alle Zeichen

Unicode (1991) definiert einen einheitlichen Zeichensatz für praktisch alle Schriftsysteme der Welt. Aktuell über 140.000 Zeichen. Jedes Zeichen hat einen eindeutigen Codepoint (Codepunkt), geschrieben als U+XXXX.

Unicode CodepointsA = U+0041 -- Lateinisches A (identisch mit ASCII) ä = U+00E4 -- Deutsches a-Umlaut € = U+20AC -- Euro-Zeichen 中 = U+4E2D -- Chinesisches Zeichen für "Mitte" 😀 = U+1F600 -- Smiley-Emoji

UTF-8 – die effiziente Kodierung von Unicode

Unicode definiert Zeichen, aber nicht wie sie als Bytes gespeichert werden. UTF-8 (Unicode Transformation Format, 8-Bit) ist die mit Abstand verbreitetste Kodierung. Über 98% aller Webseiten nutzen UTF-8.

UTF-8 ist variabel lang: einfache Zeichen brauchen 1 Byte, seltenere brauchen bis zu 4 Bytes.

Rückwärtskompatibel mit ASCII

Alle ASCII-Zeichen (0–127) werden in UTF-8 mit genau einem Byte dargestellt – mit identischem Bitmuster. Jede reine ASCII-Datei ist deshalb automatisch gültiges UTF-8. Das ist kein Zufall, sondern bewusstes Design.

UTF-8 vs. UTF-16 vs. UTF-32

Es gibt mehrere Unicode-Kodierungen. UTF-16 nutzt mindestens 2 Bytes pro Zeichen (verwendet intern von Windows und Java). UTF-32 immer 4 Bytes pro Zeichen (viel Platz, aber einfach zu verarbeiten). UTF-8 ist für westliche Texte am effizientesten und deshalb im Web Standard.

Modul 03 · Textdateien

CR, LF & CRLF

Ein unsichtbares Zeichen das regulär Produktionssysteme lahmlegt: der Zeilenumbruch. Je nach Betriebssystem wird er anders kodiert.

Die Schreibmaschinen-Herkunft

Die Begriffe CR und LF kommen aus der Zeit der mechanischen Schreibmaschinen und frühen Fernschreiber (Teletype = Fernschreiber):

CR – Carriage Return

Wagenrücklauf (0x0D)

Bei einer Schreibmaschine hatte der Schreibkopf einen "Wagen" (Carriage = Wagen) der sich nach rechts bewegte. CR bedeutete: Wagen zurück an den linken Rand. Nur horizontale Bewegung, keine neue Zeile.

LF – Line Feed

Zeilenvorschub (0x0A)

LF bedeutete: Papier um eine Zeile nach oben schieben (Line Feed = Zeilenvorschub). Nur vertikale Bewegung, kein Zurücksetzen an den Anfang.

Für einen vollständigen Zeilenumbruch bei Schreibmaschinen brauchte man also beide Aktionen: erst CR (zurück an den Rand), dann LF (eine Zeile runter). Genau das haben die ersten Computer-Systeme übernommen.

Warum hat Windows CRLF gewählt?

Die frühen Windows-Systeme basierten auf MS-DOS, das von CP/M beeinflusst wurde. CP/M war für Fernschreiber (Teletype) ausgelegt und übernahm die CRLF-Konvention direkt. Linux und Unix entschieden sich für das effizientere LF allein. Diese Entscheidung von ca. 1980 verursacht bis heute Probleme.

Wann wird das zum Problem?

Solange du nur auf einem System arbeitest, merkst du nichts. Es wird problematisch wenn Dateien zwischen Systemen ausgetauscht werden:

Shell-Skript Problem

^M am Zeilenende

Ein unter Windows bearbeitetes Shell-Skript (.sh) läuft unter Linux nicht. Der Interpreter sieht das CR (0x0D) als Teil des Befehls. Im Terminal erscheint ^M am Zeilenende.

Git-Probleme

Riesige Diffs

Git zeigt bei jeder Zeile eine Änderung an obwohl nur Zeilenenden konvertiert wurden. Lösung: .gitattributes Datei.

Modul 04 · Binärdateien

Binärdateien & Magic Bytes

Eine Binärdatei speichert Daten in einem Format das direkt für Maschinen optimiert ist. Jedes Byte kann jeden Wert von 0 bis 255 annehmen – nicht nur druckbare Zeichen.

Binär vs. Text

Textdatei

Lesbare Zeichen

Bytes repräsentieren druckbare Zeichen gemäß einer Kodierung (ASCII, UTF-8). Mit jedem Texteditor lesbar. Zeilenenden haben Bedeutung.

Binärdatei

Rohe Bytes

Bytes können beliebige Werte haben. Bedeutung hängt vom Format ab. Im Texteditor sieht man nur Zeichensalat. Braucht ein spezielles Programm.

Was passiert wenn man eine Binärdatei im Texteditor öffnet?

Der Editor versucht die Bytes als Zeichen zu interpretieren. Bytes die keine gültigen UTF-8-Sequenzen sind, werden als � (Ersatzzeichen) oder als Zeichensalat dargestellt. Die Datei selbst wird dabei nicht beschädigt – solange man sie nicht speichert.

Magic Bytes – der Fingerabdruck einer Datei

Jedes Binärformat hat eine eindeutige Signatur in den ersten Bytes: die sogenannten Magic Bytes (auch File Signature = Dateisignatur oder Magic Number = magische Zahl). Das Betriebssystem und Programme nutzen sie um den echten Dateityp zu erkennen – unabhängig von der Dateiendung.

Warum "Magic Bytes"?

Der Begriff kommt aus der Unix-Welt. Das Programm file nutzt eine Datenbank von Signaturen um Dateitypen zu identifizieren. Die Bytes wirkten für frühe Entwickler wie Magie weil sie den Typ verrieten ohne die Endung zu kennen.

Modul 05 · Dateiformate

Komprimierung

Komprimierung reduziert die Dateigröße. Es gibt zwei grundlegend verschiedene Ansätze: verlustfrei und verlustbehaftet. Die Wahl hängt davon ab was wichtiger ist – Qualität oder Größe.

Verlustfreie Komprimierung

Bei verlustfreier Komprimierung (Lossless Compression) kann die Originaldatei exakt wiederhergestellt werden. Kein einziges Bit geht verloren.

Wie funktioniert das?

Verlustfreie Algorithmen suchen nach Mustern und Wiederholungen. Statt "AAAAAABBB" zu speichern, speichern sie "6A3B" (6 mal A, 3 mal B). Das nennt sich Run-Length Encoding (Lauflängenkodierung). Modernere Algorithmen (Huffman, LZ77, LZMA) sind deutlich komplexer aber das Prinzip ist dasselbe.

Verlustfrei – Einsatz

Wenn Originalqualität Pflicht ist

Dokumente (ZIP, 7z), Programmcode, medizinische Bilder, Audio-Archivierung (FLAC), PNG-Grafiken mit Text. Alles wo ein verändertes Bit einen Unterschied macht.

Verlustfrei – Formate

PNG, FLAC, ZIP, 7z, GIF, WebP (verlustfrei)

Typische Komprimierungsraten: 20–80% kleinere Dateien je nach Inhalt. Textdateien lassen sich sehr gut komprimieren, bereits komprimierte Dateien kaum noch.

Verlustbehaftete Komprimierung

Bei verlustbehafteter Komprimierung (Lossy Compression) werden Informationen die das menschliche Auge oder Ohr kaum wahrnimmt dauerhaft gelöscht. Die Originaldatei kann nicht mehr exakt wiederhergestellt werden.

Wie funktioniert das bei JPEG?

JPEG teilt das Bild in 8x8-Pixel-Blöcke und wendet eine mathematische Transformation (DCT = Discrete Cosine Transform = Diskrete Kosinustransformation) an. Dann werden Details die das Auge schwer wahrnimmt (hohe Frequenzen, subtile Farbübergänge) verworfen. Je höher die Komprimierung, desto mehr Details gehen verloren. Typische Erkennungszeichen: blockige Artefakte besonders an Kanten.

Verlustbehaftet – Einsatz

Wenn Größe wichtiger als Perfektion ist

Fotos im Web (JPEG), Musik-Streaming (MP3, AAC), Video-Streaming (H.264). Der Qualitätsverlust ist bei guten Einstellungen für den Durchschnittsmenschen kaum wahrnehmbar.

Verlustbehaftet – Warnung

Niemals mehrfach komprimieren

Jedes Mal wenn du eine JPEG-Datei speicherst und dabei neu komprimierst, gehen mehr Details verloren. "Generation Loss" nennt man diesen Effekt. Immer im Originalformat arbeiten, erst am Ende komprimieren.

Deployment-Formate

Als FiSi wirst du Betriebssysteme und Software auf viele Rechner gleichzeitig ausrollen. Dafür gibt es spezielle Formate die du kennen musst.

ISO – das Abbild eines Datenträgers

Eine ISO-Datei (benannt nach dem Standard ISO 9660 für optische Datenträger) ist ein exaktes Abbild (Image) eines gesamten Datenträgers oder Dateisystems, Byte für Byte. Sie enthält nicht nur die Dateien selbst, sondern auch die komplette Struktur des Datenträgers.

Einsatz

Betriebssystem-Installation

Windows, Linux, macOS werden als ISO verteilt. Du lädst die ISO herunter, schreibst sie auf einen USB-Stick (z.B. mit Rufus oder dd) und bootest davon.

Einsatz

Virtuelle Maschinen

VMware, VirtualBox und Hyper-V können ISO-Dateien direkt als virtuelles DVD-Laufwerk einbinden. Kein physischer Datenträger nötig.

Einsatz

PXE-Boot

Beim Netzwerk-Deployment über PXE (Preboot Execution Environment = Vorstart-Ausführungsumgebung) wird oft eine ISO oder deren Inhalt über das Netzwerk bereitgestellt.

Linux Terminal# ISO auf USB-Stick schreiben (dd = disk dump) dd if=windows11.iso of=/dev/sdb bs=4M status=progress # if = input file, of = output file, bs = Blockgröße # ACHTUNG: /dev/sdb wird komplett überschrieben! # ISO mounten um Inhalt anzuschauen (ohne sie zu entpacken) mount -o loop windows11.iso /mnt/iso # -o loop = Loop-Device, behandelt die Datei wie ein Gerät

WIM – Windows Imaging Format

WIM (Windows Imaging Format = Windows-Abbildformat) ist das von Microsoft entwickelte Format für Betriebssystem-Images. Es wurde speziell für das Deployment von Windows entwickelt und steckt im Kern jeder Windows-Installation.

Warum WIM statt ISO?

Eine ISO ist ein Abbild eines Datenträgers. WIM ist ein Abbild eines Dateisystems. Der Unterschied: WIM ist hardware-unabhängig, komprimiert, und mehrere Windows-Editionen (Home, Pro, Enterprise) können in einer einzigen WIM-Datei stecken. Bei der Installation wählst du dann welche Edition du ausrollen willst.

install.wim / install.esd

Das eigentliche Windows-Image

Auf jedem Windows-Installationsmedium liegt unter sources\install.wim das komprimierte Windows-Dateisystem. Die .esd-Variante ist stärker komprimiert.

boot.wim

Das Boot-Image

Enthält die Windows PE (Preinstallation Environment = Vorinstallationsumgebung): ein minimales Windows das nur für die Installation gestartet wird.

Windows CMD / PowerShell – DISM# DISM = Deployment Image Servicing and Management Tool # Inhalt einer WIM-Datei anzeigen (welche Editionen sind drin) dism /Get-WimInfo /WimFile:install.wim # WIM-Image auf ein Laufwerk anwenden (auspacken) dism /Apply-Image /ImageFile:install.wim /Index:1 /ApplyDir:D:\ # WIM-Image mounten um Dateien anzuschauen dism /Mount-Image /ImageFile:install.wim /Index:1 /MountDir:C:\mount

Modul 06 · Berechtigungen

Linux Dateiberechtigungen

Unter Linux hat jede Datei und jedes Verzeichnis ein Berechtigungssystem. Es legt fest wer lesen, schreiben und ausführen darf.

Das rwx-System

Jede Datei hat drei Berechtigungsgruppen, jede mit drei möglichen Rechten:

r – read

Lesen

Datei: Inhalt lesen. Verzeichnis: Inhalt auflisten (ls).

w – write

Schreiben

Datei: Inhalt ändern. Verzeichnis: Dateien erstellen/löschen.

x – execute

Ausführen

Datei: Als Programm starten. Verzeichnis: Betreten (cd).

Diese drei Rechte werden für drei Gruppen definiert:

Owner

Eigentümer

Der Benutzer dem die Datei gehört. Ersteller einer Datei wird automatisch Eigentümer.

Group

Gruppe

Eine Benutzergruppe. Alle Mitglieder teilen diese Rechte. Nützlich für Teams.

Others

Alle anderen

Jeder der weder Eigentümer noch Gruppenmitglied ist.

Berechtigungen lesen

Mit ls -l siehst du die Berechtigungen:

Linux Terminal# ls -l Ausgabe -rwxr-xr-- 1 ben users 4096 Mai 17 datei.txt ^ ^ ^ ^ | | | | | | | +-- Others: r-- = nur lesen | | +------- Group: r-x = lesen + ausführen | +------------ Owner: rwx = alles +---------------- Dateityp: - = Datei, d = Verzeichnis

Oktalnotation

Berechtigungen können als dreistellige Oktalzahl (Basis 8) angegeben werden. Jedes Recht hat einen Wert:

chmod 777 ist fast immer falsch

chmod 777 gibt jedem Benutzer auf dem System alle Rechte. Das ist ein Sicherheitsrisiko. Typische sichere Werte: 644 für Dateien (Owner schreibt, alle lesen), 755 für Verzeichnisse und ausführbare Dateien.

Modul 06 · Berechtigungen

Windows Dateiberechtigungen & ACL

Windows nutzt ein detaillierteres Berechtigungssystem als Linux: ACL (Access Control List = Zugriffssteuerungsliste). Statt 3 Gruppen mit 3 Rechten gibt es beliebig viele Einträge mit feingranularen Rechten.

Was ist eine ACL?

Eine ACL ist eine Liste von ACEs (Access Control Entries = Zugriffssteuerungseinträge). Jeder Eintrag definiert für einen Benutzer oder eine Gruppe welche Aktionen erlaubt oder verboten sind.

Lesen

Read

Dateiinhalt und Attribute lesen.

Schreiben

Write

Inhalt ändern, Attribute setzen.

Ausführen

Execute

Programm oder Skript starten.

Löschen

Delete

Datei oder Ordner löschen.

Vollzugriff

Full Control

Alle Rechte inkl. Berechtigungen ändern.

Verweigern

Deny

Explizites Verbot überschreibt erlaubte Rechte.

Vererbung

Windows-Berechtigungen werden vererbt: Dateien in einem Ordner erben standardmäßig die Berechtigungen des Ordners. Das vereinfacht die Verwaltung, kann aber zu unerwartetem Verhalten führen wenn Berechtigungen explizit gesetzt werden.

Modul 07 · MIME & Base64

MIME-Types

MIME (Multipurpose Internet Mail Extensions = Mehrzweck-Internet-Mail-Erweiterungen) ist ein Standard der den Inhaltstyp einer Datei für Übertragungsprotokoll beschreibt. Ursprünglich für E-Mail, heute in HTTP und überall im Web.

Warum MIME-Types?

Wenn ein Browser eine Datei von einem Webserver empfängt, weiß er nicht automatisch was er damit machen soll. Der Server sagt ihm per MIME-Type: "Das ist ein JPEG-Bild" oder "Das ist JavaScript-Code". Der Browser entscheidet dann wie er die Datei behandelt.

Format

MIME-Types haben immer das Format typ/subtyp. Beispiel: image/jpeg oder text/html.

Ein Angreifer könnte eine .exe-Datei als image/jpeg deklarieren. Server sollten deshalb den MIME-Type nicht blind vom Client übernehmen, sondern den echten Inhalt prüfen (Magic Bytes). Browser wiederum sollten den Server-MIME-Type respektieren (X-Content-Type-Options: nosniff Header).

Modul 07 · MIME & Base64

Base64

Base64 ist eine Kodierungsmethode die Binärdaten in eine Zeichenkette aus 64 druckbaren ASCII-Zeichen umwandelt. Kein Komprimieren, kein Verschlüsseln – nur umformen.

Warum Base64?

Manche Übertragungsprotokolle wurden für reinen Text entworfen. Binäre Nullbytes (0x00) können dort als "Ende der Nachricht" interpretiert werden. Steuerzeichen können die Übertragung stören. Base64 löst das: es wandelt alle Bytes in druckbare Zeichen um.

E-Mail (SMTP)

Anhänge

E-Mail-Protokolle waren ursprünglich nur für 7-Bit-ASCII. Binäre Anhänge werden als Base64 kodiert.

Web / HTTP

Data-URLs

Kleine Bilder direkt in HTML oder CSS einbetten: src="data:image/png;base64,..."

APIs / JSON

Binäre Daten in JSON

JSON kann keine Binärdaten enthalten. Bilder oder Dateien werden als Base64-String verpackt.

Tokens / Auth

JWT und ähnliches

JWT (JSON Web Token) ist Base64-kodierter JSON. Kein Schutz vor Lesen, nur sichere Übertragung.

Wie funktioniert Base64?

Base64 nimmt je 3 Bytes (24 Bits) und teilt sie in vier 6-Bit-Gruppen. Jede 6-Bit-Gruppe wird einem der 64 druckbaren Zeichen zugeordnet: A-Z, a-z, 0-9, + und /.

Beispiel: "Hi" in Base64H = 0x48 = 01001000 i = 0x69 = 01101001 (Padding) = 00000000 6-Bit-Gruppen: 010010 000110 100100 000000 = 18 = 6 = 36 = 0(=) = S = G = k = = (Padding) Ergebnis: SGk=
Base64 ist keine Verschlüsselung

Base64 kann von jedem sofort dekodiert werden. Es ist nur eine Umformung, keine Sicherheitsmethode. Ein Base64-kodiertes Passwort ist genauso unsicher wie ein Klartext-Passwort. Für echte Sicherheit braucht man Verschlüsselung (AES, RSA) oder Hashing (bcrypt, SHA-256).

Größenzunahme

Base64 macht Daten größer: aus 3 Bytes werden 4 Zeichen. Das ist eine Zunahme von ca. 33%. Eine 1 MB große Datei wird als Base64 ca. 1,33 MB groß.

Modul 08 · Netzwerk-Dateisysteme

NFS & SMB

Dateisysteme müssen nicht lokal sein. Netzwerk-Dateisysteme erlauben es, Dateien auf einem entfernten Server so einzubinden als wären sie lokal auf dem eigenen Rechner. Für FiSi ist das tägliche Realität.

Warum Netzwerk-Dateisysteme?

Zentralisierung

Ein Server, viele Clients

Dateien liegen einmal zentral auf einem Server. Alle Clients greifen auf dieselben Daten zu. Kein manuelles Synchronisieren.

Backup

Einfacher zu sichern

Daten auf einem zentralen Server können zentral gesichert werden, statt auf jedem einzelnen Client-PC.

Zugriffskontrolle

Berechtigungen zentral

Wer auf welches Verzeichnis zugreifen darf, wird auf dem Server konfiguriert, nicht auf jedem Client.

SMB – Server Message Block

SMB (Server Message Block) ist das Protokoll für Dateifreigaben in Windows-Netzwerken. Die Open-Source-Implementierung unter Linux heißt Samba. SMB läuft auf Port 445.

CIFS ist SMB

Du wirst CIFS (Common Internet File System = Gemeinsames Internet-Dateisystem) als alten Namen für SMB begegnen. Technisch ist CIFS eine ältere Version von SMB. Heute meinen beide dasselbe Protokoll. In Linux-Befehlen wird häufig noch cifs als Dateisystemtyp verwendet.

SMB-Einsatz

Windows-Netzwerke

Netzwerklaufwerke in Windows (Z:, Y:...). Active Directory-Freigaben. Druckerfreigaben. Linux-Server die Windows-Clients bedienen (via Samba).

UNC-Pfad

\\Server\Freigabe

Windows-Freigaben werden als UNC-Pfad (Universal Naming Convention = universelle Namenskonvention) angesprochen: \\fileserver\daten. Doppelter Backslash, dann Servername, dann Freigabename.

Windows & Linux – SMB-- Windows: Netzwerklaufwerk verbinden net use Z: \\fileserver\daten /persistent:yes -- Oder: Netzlaufwerk verbinden mit Anmeldedaten net use Z: \\fileserver\daten /user:domain\ben Passwort # Linux: SMB-Freigabe mounten mount -t cifs //fileserver/daten /mnt/netz -o username=ben,password=pw # In /etc/fstab eintragen für automatisches Mounten: //fileserver/daten /mnt/netz cifs username=ben,password=pw,_netdev 0 0 # _netdev = erst mounten wenn Netzwerk verfügbar

NFS – Network File System

NFS (Network File System = Netzwerk-Dateisystem) ist das Standard-Protokoll für Dateifreigaben in Unix/Linux-Netzwerken. Entwickelt von Sun Microsystems (1984). Einfacher als SMB, aber ohne native Windows-Unterstützung. NFS läuft typisch auf Port 2049.

NFS-Einsatz

Linux/Unix-Netzwerke

Home-Verzeichnisse auf Linux-Servern. Shared Storage in Server-Clustern. Docker/Kubernetes-Storage. Linux-Clients die auf Linux-Server zugreifen.

NFS-Vorteil

Transparent

Gemountete NFS-Freigaben sehen für den Benutzer genauso aus wie lokale Verzeichnisse. Kein Unterschied im Alltag.

Linux – NFS# NFS-Freigabe manuell mounten mount -t nfs fileserver:/srv/daten /mnt/nfs # fileserver:/srv/daten = Server:Pfad auf dem Server # In /etc/fstab: fileserver:/srv/daten /mnt/nfs nfs defaults,_netdev 0 0 # Auf dem NFS-Server: Freigaben anzeigen showmount -e fileserver # NFS-Exporte konfigurieren (auf dem Server, /etc/exports) /srv/daten 192.168.1.0/24(rw,sync,no_subtree_check)
Die Option _netdev

Bei Netzwerkfreigaben in der /etc/fstab setzt man die Option _netdev. Sie sagt dem System: dieses Dateisystem erst mounten, nachdem das Netzwerk verfügbar ist. Ohne sie kann der Bootvorgang hängen, weil das System versucht, eine Freigabe zu mounten, bevor überhaupt eine Netzwerkverbindung steht.

Modul 08 · Netzwerk-Dateisysteme

Block- vs. File-Storage & iSCSI

NFS und SMB liefern fertige Dateien über das Netz. Es gibt aber einen zweiten Weg: rohe Festplatten über das Netz, unterhalb des Dateisystems. Das ist Block-Storage, und das wichtigste Protokoll dafür ist iSCSI.

Zwei Ebenen, Speicher bereitzustellen

Ein Dateisystem verwaltet Blöcke (Module 2: Cluster auf dem Datenträger). Die Frage ist nur: Auf welcher Ebene reicht ein Server den Speicher über das Netz weiter, auf der Datei-Ebene oder auf der rohen Block-Ebene darunter?

File-Storage

Fertige Dateien

Der Server verwaltet das Dateisystem. Der Client fragt "gib mir datei.txt" und bekommt sie. Protokolle: SMB, NFS, auch FTP.

Block-Storage

Rohe Blöcke

Der Client bekommt nummerierte Blöcke, keine Dateien. Das Dateisystem legt er selbst an. Protokolle: iSCSI, Fibre Channel.

Alltagsvergleich

File-Storage ist ein geteilter Ordner im Büro: jeder holt fertige Dokumente raus. Block-Storage ist eine leere externe Festplatte, die dir per Kabel gereicht wird, nur dass das "Kabel" hier das Netzwerk ist. Du formatierst sie selbst und legst dein eigenes Inhaltsverzeichnis an. Der Server auf der anderen Seite weiß nicht mal, welche Dateien du darauf speicherst.

Was ist iSCSI?

iSCSI (Internet Small Computer System Interface) nimmt die Befehle, mit denen ein Betriebssystem normalerweise eine lokale Festplatte anspricht (SCSI-Befehle wie "lies Block 4711"), und schickt sie verpackt in TCP/IP-Pakete über ein normales Ethernet-Netz an einen anderen Rechner. Für den Client sieht das Ergebnis aus wie eine ganz normale eingebaute Platte. Dass sie in Wirklichkeit im Netzwerk hängt, merkt er nicht.

Initiator

Der Client

Das System, das auf den Speicher zugreift. Es "initiiert" die Verbindung. Beispiel: ein Server, der seine Platte vom Storage holt.

Target

Das Ziel

Das System, das den Speicher bereitstellt. Das "Ziel" der Verbindung. Beispiel: ein NAS oder ein Storage-System.

LUN

Die virtuelle Platte

Logical Unit Number. Die einzelne "Festplatte", die das Target rausgibt. Oft eine große Datei oder ein Volume auf dem Storage.

IQN

Die Adresse

iSCSI Qualified Name, z.B. iqn.2026-06.com.beispiel:storage1. Eindeutiger Name, über den sich Initiator und Target ansprechen.

Wie der Zugriff abläuft

Der Initiator bekommt die IP des Targets, fragt "was hast du für mich?" (Discovery) und verbindet sich mit einer LUN. Danach taucht im Betriebssystem eine neue, leere Platte auf. Die muss erst initialisiert, partitioniert und formatiert werden (z.B. NTFS), genau wie eine frische physische Platte. iSCSI läuft über TCP auf Port 3260 und gehört aus Performance-Gründen in ein eigenes, getrenntes Netz.

NAS vs. SAN: das größere Bild

iSCSI ist ein Protokoll. NAS und SAN sind dagegen Architekturkonzepte, also Bauweisen, wie man Speicher im Netz bereitstellt. Sie liegen auf einer anderen Ebene als iSCSI, und genau hier wird oft durcheinandergeworfen.

NAS
  • Network Attached Storage
  • Liefert File-Storage
  • Protokolle: SMB, NFS
  • Client sieht eine Netzwerkfreigabe
  • Dateisystem liegt beim NAS
SAN
  • Storage Area Network
  • Liefert Block-Storage
  • Protokolle: iSCSI, Fibre Channel
  • Client sieht eine lokale Festplatte
  • Dateisystem liegt beim Client
Der Zusammenhang

iSCSI ist eines der Protokolle, mit denen man ein SAN aufbaut. iSCSI und SAN sind also keine Konkurrenten, sondern iSCSI ist das "Wie" und SAN das "Was". Wichtig: Hinter iSCSI muss nicht zwingend ein großes SAN stecken. Auch ein einzelner Server oder ein NAS kann einem anderen System per iSCSI eine Platte geben.

Faustregel

iSCSI gibt einem einzelnen System eine exklusive Platte. SMB/NFS geben vielen Nutzern einen geteilten Ordner. Sobald mehrere gleichzeitig auf dieselben Daten zugreifen sollen, ist iSCSI das falsche Werkzeug, denn zwei Systeme, die gleichzeitig auf dieselbe LUN schreiben, zerstören sich das Dateisystem. Deshalb ist iSCSI ein Server- und Infrastruktur-Thema (VM-Speicher, Datenbanken, Backup-Ziele), nicht etwas für normale Arbeitsplatz-Clients.

Für die IHK

Den einen Satz solltest du sicher können: iSCSI ist ein Block-Storage-Protokoll, mit dem ein Initiator über IP (Port 3260) auf das Target eines SAN zugreift und den bereitgestellten Speicher wie eine lokale Festplatte nutzt. Dazu die Abgrenzung Block- vs. File-Storage und die Einordnung NAS (File) gegen SAN (Block).
